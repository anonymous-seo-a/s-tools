# 次セッション用ハンドオフプロンプト

このファイルは、新規チャットセッションを開始するときに、Claudeに最初に渡すプロンプトとして使用する。
これにより、Claudeはこのプロジェクトの現状を即座に把握し、設計を継続できる。

---

## 使い方

新規チャットを開始したら、以下のテキストブロックを最初のメッセージとしてコピペで送る。

---

## ハンドオフプロンプト本文（以下をコピペ）

```
このセッションは、Daikiの「自走リライトシステム」の設計を継続するためのものです。

# 現在地
- Phase 2 の Step A（最優先5手法の採用判定）は完了
- 次に進む作業: 案E（既存スコアリング統合仕様の確定）

# 設計確定事項
プロジェクトのナレッジファイル knowledge_05_rewrite_system_design.md
に Step A-1 〜 A-4 の確定事項が全て記録されている。
新規テーブルのSQL、採用したインフラ、却下した選択肢の理由などが含まれる。
このファイルを必ず参照してから設計を進めること。

# 進行順序
1. 案E: 既存スコアリング統合仕様の確定（次に進む）
   - GitHubリポジトリ anonymous-seo-a/s-tools の確認が必要
   - 既存記事スコアリングのインプット・アウトプット・ロジック・運用方法
   - Information Gain Score / Microsoft Clarity との統合方法
2. 案C: 工程6'（LLM実行レイヤー）の設計
3. 案D: Step A-1〜A-4 の統合確認
4. 案B: 強推奨手法の確定（最優先5以外）
5. Phase 3: 詳細設計
6. Phase 4: 実装

# 設計原則
- 真=美フィルタの3軸（必然性・閉合性・最小性）を全判断に適用
- 遅延評価（lazy evaluation）型処理
- 抽象化レイヤー（API差し替え可能）
- 3層学習ループ（記録 → 抽出 → 反映）
- 主指標 × Clarity gating の二段判定
- ポジトーク回避（Google一次情報ベース）

# 進行スタイル
- 設計判断ごとにステップを分解して、私（Daiki）に判断を仰ぐ
- インプット・アウトプットを明確に定めてパーツごとに動くように設計する
- 出典の質を厳守（Google Search Central > 業界専門家複数支持 > 個別ベンダー単独支持）
- 真=美の最小性テストに反する設計は提示しない（過剰な機能追加を避ける）

# 文体
- 結論先行、構造的、端的
- 根拠のない励まし、美辞麗句、冗長な前置きは余剰として削る
- 構造を示すときは図・箇条書き・コードブロックを優先
- 反証プロトコル: 主張に対する逆方向検証を必ず付記

# 会話切れ対策
このセッションでも、Step確定ごとに knowledge_05 を更新する想定。
セッション終了時には、次セッション用のハンドオフプロンプトを更新する。

それでは案E（既存スコアリング統合仕様の確定）から進めてください。
最初のステップとして、GitHubリポジトリ anonymous-seo-a/s-tools 
の構造を私から共有する方法を提案してください。
```

## 補足: ハンドオフプロンプトの更新タイミング

このプロンプトは Step を進めるごとに陳腐化する。以下のタイミングで Daiki が更新する：

- 案E が完了したら、「次の作業」を 案C に変更
- 案C が完了したら、「次の作業」を 案D に変更
- 各案完了時に、「現在地」「進行順序」を更新

更新内容は、各セッション終了時に Claude が新しいハンドオフプロンプトを生成するので、
それを使ってこのファイルを上書きする運用を推奨。

---

## 補足: ナレッジファイルとハンドオフプロンプトの役割の違い

| ファイル | 役割 | 更新頻度 |
|---|---|---|
| knowledge_05_rewrite_system_design.md | 設計確定事項の構造化保存 | Step確定ごと |
| handoff_prompt_for_next_session.md | 新規セッション継続用の起動プロンプト | Step進行で「次の作業」が変わるごと |

両者は補完関係：
- ナレッジファイルは「何が確定したか」の参照先
- ハンドオフプロンプトは「次に何をするか」の起動装置
