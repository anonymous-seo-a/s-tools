# 真=美フレームワーク（設計原理）

> 「美しさは審美的好みではなく、構造的必然性である」

本システムの全判断はこの原理に基づく。

## 3 つの判定テスト

| テスト | 問い |
|--------|------|
| **必然性テスト** | この要素を除いたら構造は崩壊するか |
| **閉合性テスト** | 外部への依存を最小にできているか |
| **最小性テスト** | より少ない要素で同じ機能を実現できないか |

これらは記事品質の判定にもコード設計にも適用する。

## システム設計の重心

```
誤った設計: LLM にリライトさせる
正しい設計: マスターデータを主役にし、LLM は検出+組立+生成を担当
```

### 理由

- LLM の強み = **生成**（テキスト量産）
- LLM の弱み = **判定**（何が必然で何が余剰か）
- Daiki の強み = **判定**（真=美の検出）

LLM に「いい感じにリライトして」と任せた瞬間、出力は一般的 AI 記事に収束し、
soico /no1/ の CVR 5〜10% は再現されない。

属人的な真=美の判定基準を**マスターデータに外部化**することで、
LLM は決定論的な「検出 → マスター参照 → 配置」のパイプラインに専念できる。

## 役割分担

| 役割 | 担当 |
|------|------|
| 真=美の判定 | Daiki（委譲不可） |
| マスターデータ構築 | ゆかちゃん（Daiki検証必須） |
| 検出 + 組立 + 生成 | LLM (Claude API) |
| 実装 | Claude Code |
| 最終承認 | Daiki |

## YMYL 記事の品質基準: 検索意図 3 層

- **顕在ニーズ**: 検索者が言語化している表層のニーズ
- **潜在ニーズ**: 顕在の背後にある動機（バレたくない、無職、等）
- **安心ニーズ**: YMYL 信頼性（違法業者でないか、法的に問題ないか）

3 層全てが「閉じている」記事のみが Google に 1 位として評価される。

## 2026 年環境の追加要素

- **AI 引用適性**: AI Overview に拾われる設計（冒頭 40-60 語直接回答、質問見出し、FAQ 構造）
- **Experience**: 一次情報（体験、独自調査、独自画像）の重要性が最大化

## 4 つの技術課題と統一的解決

注釈・画像・アフィリエイトリンク・出典リンクは全て「真=美の必然要素」として統一的に扱える。

```
記事 = テキスト + 注釈 + 画像 + アフィリンク + 出典リンク
全要素が真=美テストを通過した状態 = 完成
```

各要素はマスターテーブルとして外部化し、LLM は「検出 → マスター参照 → 配置」のパイプラインで処理する。

## 記事品質の必須 16 要素

| # | カテゴリ | 要素 |
|---|---------|------|
| 1-4 | 検索意図閉合 | 顕在 / 潜在 / 安心ニーズ応答、AI 引用適性 |
| 5-8 | E-E-A-T | 一次情報、著者、監修者、鮮度 |
| 9-11 | 構造 | 結論先行、見出し階層、内部リンク |
| 12-14 | テクニカル | Core Web Vitals、構造化データ、モバイル |
| 15-16 | 差別化 | 実務詳細、独自構造化要素（比較表/マトリクス/決定木/スコア診断） |

## 反証プロトコル

各モジュール実装後、Daiki が「これは違う」と言った場合は、
直感を正とみなし**前提から再構築**する。

仕様書通りの実装が必ずしも正解ではない。設計者の直感が現場を最も知っている。
