# プロジェクト: SEOリライトツール開発（真=美フレームワークベース）

> 原本 2026-04-24 Daiki 提供。統合先は s-tools（本リポジトリ `/var/www/rewrite`）。Phase 0 協議で合意した修正を §変更履歴 に記録。

## 1. プロジェクトの目的

既存 SEO 記事を AI が自動リライトし、人間レビュー後に WordPress へ反映するツールを構築する。原案は「リライト指示書出力」止まりだったが、Phase 0 協議で **AI が実リライトまで実行**する方針に変更（指示書は内部中間表現として保持、UI でも併記）。

既存運用の問題:
- SEO判断が属人的で再現性がない
- リライト対象選定が経験依存
- 金融YMYL領域の品質基準が明文化されていない
- AI Overview時代の環境変化に追従できていない

解決の方向性:
- 判定ルーチンをテスト群に分解して外部化
- 複数データソースを統合して客観データベースで判定
- 人間介入が必要な領域（HumanFlag）を明示的に設計

## 2. 前提フレームワーク

本ツールの判定基盤は「真=美フレームワーク」に基づく。

**真=美の定義**: 美しさは審美的好みではなく、構造的必然性である。必要十分で自己完結した最小の閉じた構造。

3つの判定テスト:
- **必然性テスト**: この要素を除いたら構造は崩壊するか
- **閉合性テスト**: 外部への依存を最小にできているか
- **最小性テスト**: より少ない要素で同じ機能を実現できないか

SEO領域での検索意図3層構造:
- **顕在ニーズ**: 即答欲求
- **潜在ニーズ**: 背後の動機
- **安心ニーズ**: YMYL信頼性

2026年環境の追加2要素:
- **AI引用適性**: AI Overviewに拾われる設計（冒頭40-60語の直接回答、質問見出し、FAQ構造）
- **Experience**: 一次情報（体験、独自調査、独自画像）の重要性が最大化

## 3. 完成仕様

### 3.1 記事品質判定の必須16要素

【検索意図の閉合】
1. 顕在ニーズ応答
2. 潜在ニーズ応答
3. 安心ニーズ応答（YMYL信頼性、法令引用を内包）
4. AI引用適性（冒頭直接回答、質問見出し、FAQ構造を内包）

【E-E-A-T】
5. 一次情報（独自データ、独自画像、体験を内包）
6. 著者情報
7. 監修者情報
8. 鮮度（公開日、更新日、情報の最新性）

【構造的要素】
9. 結論先行
10. 見出し階層（H1-H3、キーワード配置）
11. 内部リンク（トピッククラスター）

【テクニカル】
12. Core Web Vitals（記事単位対策範囲: 画像最適化、レイアウト安定化）
13. 構造化データ（Article/FAQPage/BreadcrumbList/Person）
14. モバイル適性

【差別化】
15. 実務詳細（競合省略箇所）
16. 独自構造化要素（比較表、マトリクス、決定木、スコア診断）

### 3.2 処理モジュール（修正後: 9モジュール）

> **Phase 0 協議で修正**: 原案の [3]必然性 / [4]閉合性 / [5]最小性 を 1 モジュールに統合（3観点スコアを返す形）。パターン別ルーティング (3.3) は廃止し全モジュール並列実行。

```
[0] リライト対象選定       ← s-tools SQLite (daily_metrics, daily_scraped_rank 等)
[1] SERP分析              ← Ahrefs CSV アップロード（手動、API は Phase 後半）
[2] 検索意図3層分解       ← LLM
[3] 真=美判定             ← 必然性 / 閉合性 / 最小性を 1 呼び出しで 3 観点評価
[4] E-E-A-T充足度         ← LLM + WP 記事メタ
[5] AI Overview適性       ← LLM（冒頭直接回答・質問見出し・FAQ検出）
[6] テクニカル要素        ← HTML解析（Article/FAQPage schema、画像、見出し階層）
[7] リライト指示書生成    ← 全モジュール結果を集約、優先度スコアリング
[8] ファクトチェック      ← e-Gov API等（YMYL特化、Phase 後半）
```

### 3.3 全モジュール並列実行（旧: 悪化パターン別ルーティング）

> **Phase 0 協議で修正**: 原案の「A/B/C/D パターン分岐」は廃止。全対象記事に全モジュール並列実行し、出力側で fail テストの重み付けで優先度を提示する。

モジュール [0] は依然として悪化種別（順位 / CTR / エンゲージメント / 複合）のタグ付けを行うが、分岐制御ではなく出力の見出しとして使う。

### 3.4 データソース統合

| データソース | 役割 | Phase |
|------------|------|------|
| s-tools SQLite（本リポジトリ内） | 記事一覧・順位・PV・アフィCR・GSC・apply_history | Phase 1〜 |
| Ahrefs CSV アップロード | 外部 SERP / 競合 / 被リンク | Phase 2〜（API 化は後発） |
| Microsoft Clarity | UX（スクロール、離脱、クリック） | Phase 3〜（初期は手動 CSV） |
| e-Gov等公的API | ファクトチェック（法令条文照合） | Phase 後半（YMYL 強化時） |

### 3.5 真=美テスト群の自動化方針

各サブタスクは以下の分類で実装:
- **完全自動化**: ルールベース、embedding、APIコール
- **半自動化**: LLM推論 + 信頼度スコア + HumanFlag
- **HumanFlag**: 低信頼度判定は人間介入で保留

属人的判断の外部化度目標:
- 必然性テスト: 70-80%
- 閉合性テスト: 60-70%
- 最小性テスト: 80-90%
- 全体平均: 70-80%（残りはHumanFlag運用）

### 3.6 出力仕様

#### 3.6.1 内部表現（指示書 3 層）

- **戦略層**: なぜこの構成か（SERP 分析、差別化軸、コア戦略）
- **構成層**: H2/H3 一覧、セクション別 Keep/Update/Add/Remove
- **実行層**: 書くこと/書かないこと/必要データ/出典 URL + HumanFlag（一次情報投入箇所）

`rewrite_instructions` テーブルに正規化し、`layer` 列（strategy/structure/execution）で分類。

#### 3.6.2 AI リライト実行

指示書をもとに AI（`claude-opus-4-7`）が記事本文を直接リライトし、`rewrite_drafts` テーブルに保存。HumanFlag 付きの箇所は `{{HUMANFLAG: 一次情報投入}}` 等のプレースホルダで出力し、レビュー UI で人間が埋める。

#### 3.6.3 レビュー UI（事前確認）

- **Before / After diff**: 段落単位で追加=緑 / 削除=赤 / 変更=黄にハイライト
- 横に戦略層/構成層/実行層を表示して「なぜこの修正か」を併記
- HumanFlag 箇所は赤枠で明示、そこだけ手動入力欄
- 承認ボタン押下で WordPress へ反映、却下で破棄、書き換えでドラフト保存

#### 3.6.4 WordPress 書き込み

既存 `gap-fill.js updateWpPost()`（WP REST API `POST /posts/{id}` + Basic 認証 + Application Password）を流用。`save_post` フック経由で AIOSEO のサイトマップ再生成 / lastmod 更新 / Search Console ping が自動実行される前提。

書き込み直前に AIOSEO 設定を確認:
- Sitemap 設定「記事更新時に再生成」が ON
- Search Console ping 有効

本番運用は 100 本/月（≒ 3-4 本/日）なので REST API のレイテンシは許容範囲。PHP 直呼び経路は作らない（二重管理回避）。

## 4. 技術要件

### 動作環境
- Daiki側: Mac（Apple Silicon）、LocalWP、VS Code
- 本番: Xserver（PHP 7.4.30、MySQL 8.0.35、nginx）
- 対象サイト: WordPressベース（no1-soico theme）
- **統合先**: s-tools 本リポジトリ `/var/www/rewrite`（Node + Express + SQLite + React、pm2 プロセス rewrite-app）

### セキュリティ
- APIキーは環境変数管理
- Xserver本番環境への直接書き込みは避ける（リライト指示書は手動適用）
- Daiki個人のデータは外部送信しない

### LLM選定
- Claude API（Anthropic）推奨。既存 monitor-analysis.js で `claude-opus-4-7` を使用中。本ツールも同等を使用。
- 必要に応じて軽量タスクは Gemini Flash 等を併用（現時点で必然性なし）

## 5. 運用モード

- **対話的進行**: 各Phaseの開始時にDaikiの承認を得る
- **段階的実装**: モジュール単位で実装して動作検証してから次へ
- **反証プロトコル**: 各モジュール実装後、Daikiが「これは違う」と言った場合は、直感を正とみなし前提から再構築する

## 6. 制約事項

- AI生成コンテンツが「大量自動生成」と判定されないよう、Daikiの一次情報投入を前提設計とする
- リライトツール単体での自動公開は実装しない（必ず人間の最終承認を経る）
- Google品質評価ガイドライン（2025年1月改訂）に準拠

## 7. 運用スペック（Phase 0 確定）

| 項目 | 値 |
|------|----|
| スループット | 100 本以上/月（3-4 本/日） |
| 利用者 | Daiki + soico 編集部（数名） |
| 成功指標 | リライト後の順位 + PV、翌日 + 1 週間後で観測 |
| AI モデル | `claude-opus-4-7`（既存 monitor-analysis.js と同一） |
| 埋め込み | Voyage AI（無料枠 200M tokens/月）を優先検討、sqlite-vec で近傍検索 |
| WP 書き込み | REST API（既存 `updateWpPost()` 流用） |
| 承認フロー | AI 生成 → Before/After diff UI → 人間承認 → WP 反映 |
| HumanFlag | 一次情報投入・実体験記述など、人間のみ判断可能な箇所にインラインマーカー |
