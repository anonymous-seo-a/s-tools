## Phase 1: Claude Code レポート受領と初期解釈

### レポート受領

Claude Code から構造化レポートを受領。主要発見:

**3系統のスコアの存在**:
- A系（runScoring / scoring.js）: CTA挿入優先度
  - 式: `(gscClicks × posCoeff) / (affiliateClicks + 1)`
  - 永続化: メモリのみ
- B系（alertScore / MonitorView.jsx）: 順位観測アラート
  - 直近変動の検出
  - 永続化: フロント計算のみ
- C系（monitor-analysis.js）: Claude opus による要因分析
  - 自然言語コメント
  - analysis_comments テーブルに永続化

**重要な発見（私の初期解釈）**:
1. A系とIG Scoreは直交軸（ビジネス軸 × コンテンツ軸）
2. C系の buildPrompt が案Cの設計を半分終わらせている（70-80%完成相当）
3. monitor-collectors.js / monitor-db.js は再利用可能な高品質資産

### 私の初期推奨（後で誤りと判明）

```
論点1（対象選定の統合方式）への私の推奨:
  選択肢④（A系+B系+新C補完）+ 案β（独立キュー）

  キュー2: B系（直近悪化アラート）
  キュー3: 新C（CTR異常、A系の盲点補完）
```

### 重要な軌道修正1: 既存スコアの目的取り違え（最大の修正）

**Daiki の指摘**:

> 「元々こちらの評価システムはCTA挿入と順位観測のために作ったシステムだ。そもそもリライトすべき記事のピックはロジックが違うので、別軸で作り直した方が良いと思うのですがどう思われますか？」

私の自己検証:

```
私が見誤っていた構造:
  既存スコア = 「リライト対象選定にも転用可能」と誤解
  
本当の構造:
  A系の式 = 「CTA挿入優先度」として完全に閉じている
    → 「機会はあるがアフィクリック取れてない」を捉える
    → リライトとは別目的
    
  B系の式 = 「順位観測」として完全に閉じている
    → 急落の早期検出
    → リライトとは別目的
    
  リライト対象選定の本質 = 
    「現在の記事が、現在の検索意図に対して構造的不足を抱えているか」
    → 既存A系/B系では一切捉えられていない軸
```

→ **案Eの設計方針を根本転換**:
- 旧案E: 既存スコアリングを統合する仕様の確定
- 新案E: リライト対象選定ロジックの新規設計

### 学び: 私の盲点（既存資産への過剰適応バイアス）

```

正しい処理パターン:
  既存資産が高品質 → 目的が一致するか? → Yes なら流用、No なら別設計
```

Daiki の認知（直感「ロジックが違う」）が補完した。

### 論点2': 既存システムとの役割分離

#### サブ論点

```
2'-a: データ層の共有方針
2'-b: スコア層の独立方針
2'-c: 効果測定の統合方針
2'-d: 物理的配置（DB分離 / DB共有）
```

#### 物理配置の選択肢

**案A: 同一DB（monitor.db を拡張）**
- 利点: JOIN直接可能、運用単一化
- 問題: 責任境界混濁、障害分離不可

**案B: 別DB（rewrite.db 新設）**
- 利点: 責任境界明確、障害分離、独立バージョン管理
- 問題: ATTACH DATABASE のオーバーヘッド（軽微）

#### 私の推奨と Daiki の判定

私の推奨: 案B（別DB）  
Daiki の判定: **案B**

理由:
1. 真=美の閉合性: 既存システムとリライトシステムは別目的・別責任
2. 障害分離: リライトシステムの不具合が既存システムに波及しない
3. SQLite の ATTACH DATABASE は標準機能、性能ペナルティ実質ゼロ

#### Read-Only 強制の実装方針

```
SQLite URI: file:/path/to/monitor.db?mode=ro
→ コードレベルで Read-Only を技術的に保証
```

## 採用却下した選択肢の詳細理由

### 既存スコアリング（A系/B系）のリライト対象選定への流用
- **理由**: 目的が違う（CTA挿入優先度・順位観測 vs リライト対象選定）
- 流用は構造的混濁を生む
- 既存スコアの式（A系: `(gscClicks × posCoeff) / (affiliateClicks + 1)`）は CTA挿入として完全に閉じている。リライトとは別問題

### 軸4（構造的健全性）を対象選定軸として採用
- **理由**: 個別記事選定とサイト全体監査は別レイヤー
- 混在は閉合性違反
- Site Reputation Abuse 検出は別レイヤーで扱う方が真=美

### 合算スコア方式（軸を加重和）
- **理由**: 重みの決定根拠が乏しい（恣意的）
- 軸の意味が消える → Daikiが「なぜこの記事が上位か」を逆算できない
- 真=美の閉合性違反

### 同一DB（monitor.db 拡張）方式
- **理由**: 責任境界が混濁
- 既存システムとリライトシステムが密結合
- 障害分離不可

### 案E初期推奨（A系+B系+新C 独立キュー）
- **理由**: 既存スコアの目的を取り違えていた
- Daikiの指摘により転換

### 軸2/3の合算
- **理由**: 経済合理性（軸2）と鮮度（軸3）は独立した次元
- 独立キューで両者の意味を保存する方が真=美

## 学び（このセッションを通じて確立されたもの）

### 1. 「既存資産の流用」は目的が一致している場合のみ真=美

既存スコアリングは技術的には流用可能だが、目的が違うと構造的混濁を生む。
Daikiの直感（「ロジックが違う」）が正しかった。

```
既存資産の評価フロー:
  1. 技術的に流用可能か?
  2. 目的が一致しているか? ← ここが本質
  3. Yes なら流用、No なら別設計
```

私は1で止まっていた。Daikiが2で正した。

### 2. 私の盲点パターン: 既存資産への過剰適応バイアス

レポートを精読した結果、私は「既存資産の構造的把握」に集中し、「目的の違い」という本質を見落とした。

```
私の処理傾向:
  既存資産が高品質 → 流用したい → 流用できる前提で組み立て

正しい処理:
  既存資産が高品質 → 目的が一致するか? を最初に問う
```

このバイアスは案Eだけでなく、今後の設計でも警戒対象。

### 3. 真=美の閉合性テストの実適用パターン（軸4分離）

軸4が「対象選定」と「サイト全体監査」の2レイヤーに跨ることを、必然性テストで検出。

```
軸4の混入が生む問題:
  個別記事選定の判断軸に「サイト全体の構造」が混在
  → 個別記事のスコアと全体監査のスコアが同一テーブルに同居
  → 責任境界が崩れる
  
レイヤー分離が真=美:
  対象選定 = 個別記事の判定
  サイト監査 = 全体構造の判定
  両者は別レイヤーで実装、別タイミングで実行
```

### 4. 認知リソース配分の最適化

Daikiの「Claude Codeに探させる」判断は、認知リソース配分の最適化として真=美。

```
従来想定:
  Daiki が手動探索 → 私が解釈
  
最適化後:
  Claude Code が探索 → 私が解釈 → Daikiは判断のみ
  
Daiki の認知リソースは「真=美判定」に集中
私の認知リソースは「構造設計」に集中
Claude Code の処理能力は「探索」に集中
```

役割分担の構造が真=美。

### 5. 段階的MVP戦略の有効性

3 Phase に分解することで、各Phase 完了時点で運用開始可能な状態を作る。

```
Phase 1: 軸2 + 軸3経過月数 → 単独運用可能（既存データのみ）
Phase 2: 軸1追加 → 自走システム本格稼働
Phase 3: 軸3拡張 → YMYL最重要対策の自動化完了

各Phaseが独立に価値を生む
失敗してもロールバックが容易
```

### 6. 反証プロトコルの有効性

各論点で反証を提示することで、想定外の論点が浮上する。

このセッションでも以下が反証で確認された:
- 「Read-Only 参照は本当に守れるか?」 → SQLite URI で技術的保証
- 「DB分離だとトランザクション境界は?」 → 同一トランザクションは不要
- 「20日の工数見積もりは楽観的では?」 → バッファ含めて 26〜30日に修正
- 「並行作業は本当に可能か?」 → 認知容量制約を考慮し時系列分離が現実的
