## 論点1: テーブル統合確認

### 議論の出発点

全テーブルの関係図を作成し、FOREIGN KEY の解決可能性を確認。

### 重要な発見1: ATTACH 越し FK の構造的不整合

Step A-1 / A-3 の SQL 定義を確認したところ:

```sql
FOREIGN KEY (article_id) REFERENCES articles(id)
```

これは案E 確定方針「monitor.db を ATTACH DATABASE で Read-Only 参照」と衝突。SQLite の FK 制約は同一 DB 内でのみ機能し、ATTACH 越しの参照は宣言できても**強制されない**。

影響範囲:
- master_fact_set
- master_information_gain_score
- master_evidence_article_link
- 案E のテーブルは既に FK 宣言なしで post_id 持ち（整合）

### 重要な発見2: 識別子の名称不統一

Step A 系では `article_id`、案E では `post_id` を使用。monitor.db.articles の主キーは案E で `m.articles.post_id` と明記されている。
→ Step A 系の article_id は実体としては monitor.db.articles.post_id を指していた。命名乖離は将来の保守性を下げる。

### Daiki の判定（論点1）

```
1.A: A1 (FK 宣言を全削除、論理参照に統一、コメント明示)
1.B: B1 (識別子を post_id に統一)
1.D: D1 (master_fact_set ↔ master_evidence の二経路維持)
```

理由:
- A1: 宣言と実態の一致が真=美の閉合性
- B1: monitor.db が Single Source of Truth（案E 確定）
- D1: 「発生源」と「引用位置」の役割分離は意味論的に正しい

### 論点1.C: 案C テーブル SQL の確定

5サブ論点で確定:

```
1.C-α: α1 (change_type 9種 + change_category 8種、両方保持)
       理由: LLM 出力の操作と学習用分類の役割分離

1.C-β: β3 (high_risk_categories は session 配列 + diff 単一フラグ両方)
       理由: 案K の判定単位（session）と diff レベルの警告表示の両立

1.C-γ: γ2 (queue_id NOT NULL、全 session が queue 経由)
       理由: 学習ループで「軸別効果分析」を可能にする
       → 後に 4.M' で多対多化して再解釈

1.C-δ: δ1 (evidence_id NULL 許容)
       理由: タイトル変更等 Evidence 不要な差分が存在

1.C-ε: ε1 (ab_test_id 後追い設定)
       理由: SearchPilot 方式（1000記事単位）と差分単位の粒度不整合
```

### 論点1.C-γ の追加解説

Daiki から「これだけちょっとわかりにくいから詳しく」との要請で、queue と session の関係を再説明:

- queue = 「対象として選定された」状態
- session = 「実際に LLM 実行した」記録
- 通常: queue → session の順で発生
- 例外: 手動緊急着手では queue を介さない可能性

3案を運用観点で比較:
- γ1 (NULL 可): 緊急対応可、追跡△
- γ2 (NOT NULL): 全リライトが軸体系に乗る、追跡完全
- γ3 (カラムなし): 学習ループで軸別効果集計が不可能

→ Daiki 判定: γ2

### 反証

**「session に2種類の判定状態を持たせるのは閉合性違反では」**:
- 方針判定（案K）と差分判定（案L）は別 LLM 呼び出しの結果に対する判定
- 2フェーズが session 内で完結するため、別テーブル化は逆に冗長
- 状態機械として status カラムが遷移を表現

**「change_type 9 種類は最小性に反する」**:
- A/B テスト change_category と1対1対応のため、追加マッピング層が必要
- 現状が最小

## 論点2: データフロー閉合性確認

### 議論の出発点

工程1〜8 の入出力連鎖の検証。各境界で「上流の出力 = 下流の入力」が成立するか。

### 重要な発見3: post_id → seed_query 変換キー不在

工程1（案E）の出力は post_id だが、工程2-3（Query Fan-out）の入力は seed_query。両者の変換ロジックが未定義だった。

3案:
- A1: monitor.db.articles の主キーワードカラム参照
- A2: GSC 最大インプレッションクエリを seed に
- A3: rewrite.db に target_query を持つ中間テーブル新設

→ Daiki 判定: A3

理由:
- monitor.db は Read-Only で、articles へのカラム追加は不可
- 「1記事に複数 target_query」「source の記録」「履歴管理」を rewrite 側責務として持つべき
- 別テーブル化は閉合性◎、最小性も合格

### 重要な発見4: master_query_fanout ↔ master_competitor_corpus が文字列マッチ

両者の接続が target_query (TEXT) の文字列一致のみだった。閉合性違反の疑い。

3案:
- B1: master_competitor_corpus に query_fanout_id を追加
- B2: 文字列マッチで運用
- B3: マッピングテーブル化

→ Daiki 判定: B1

### 重要な発見5: queue 緊急着手と γ2 の整合性

γ2「queue_id NOT NULL」を採用したため、緊急手動着手も必ず master_rewrite_queue 経由になる。
しかし master_rewrite_queue.selected_axis / selected_score は NOT NULL → 手動緊急着手で軸とスコアが存在しない問題。

3案:
- W1: 擬似値 'daiki_manual_priority' を許容
- W2: NULL 可に変更
- W3: 必ず軸とスコアを指定（運用ルール）

→ Daiki 判定: W2

理由:
- 緩むのは selected_axis のみ、queue_id 自体は NOT NULL → 追跡は完全
- 「手動着手」の事実は selected_by='daiki_manual' で記録される
- 学習ループでも独立カテゴリとして集計可能

### 論点2 のその他の確定

```
2.F: F1 (change_category 単位で diff を集約 → ab_test 自動生成、月次バッチ)
2.G: G2 (該当 change_category の active pattern のみ注入、上位5件)
```

### 反証

**「2.A で A3 を選ぶと冗長では」**:
- target_query を rewrite 側で管理する責務分離が真=美
- monitor.db のカラム追加は Read-Only 制約と矛盾

**「2.W で W2（NULL可）にすると γ2 の閉合性が緩む」**:
- 緩むのは selected_axis のみで、queue_id 自体は NOT NULL → 追跡は完全
- → むしろ閉合性向上

## 論点3: 真=美原則の最終確認

### 必然性テスト

全16テーブル + 全8工程について「除いたら何が崩壊するか」で検証。
→ 削減可能なものなし。

### 閉合性テスト

外部依存5系統（monitor.db / GSC / GA4 / WP / Clarity）は全て必要・代替不可。
API 依存4系統（SerpApi / Anthropic / OpenAI / Gemini）は Adapter 化済み。

### 重要な発見6: Wayback Machine が Adapter 化されていない

Step A-3 確定事項では「Wayback Machine 自動アーカイブ」となっていたが、URL検証ロジックが Wayback 限定で固定化されていた。
他の Adapter（SerpApi / LLM）と一貫性がない。

2案:
- H1: Adapter 化、複数アーカイブサービス対応
- H2: 単一サービス確定で維持

→ Daiki 判定: H1

理由:
- 中長期で切替時の二重負荷リスクを回避
- 抽象化コストは小さい

### 最小性テスト: 重複機能の検証

6項目の重複懸念を検証:

1. master_fact_set ↔ master_evidence: 役割分離（事実主張 vs 裏付け資源）→ 重複なし
2. master_ab_test_result ↔ master_clarity_signal: 主指標 vs UX 信号 → 重複なし
3. master_competitor_corpus ↔ master_query_fanout: 階層構造 vs 競合スナップショット → 重複なし
4. master_rewrite_target_score ↔ master_information_gain_score: 集計結果 vs 生データ
5. status カラム群: 各テーブルで意味が異なる → 重複なし
6. リスク管理カラム群: llm_confidence と risk_flag に重複の懸念

### 重要な発見7: llm_confidence と risk_flag='low_confidence_output' の意味重複

```
diff.llm_confidence='low' = LLM 自身が自信なしと申告
diff.risk_flag='low_confidence_output' = リスクカテゴリとして low_confidence を分類

→ 同じ事実を別カラムで表現
```

2案:
- J1: risk_flag から 'low_confidence_output' を除外
- J2: 維持

→ Daiki 判定: J1

注意: session.high_risk_categories は維持（5カテゴリ全て）。session レベルでの案K 発動条件として 'low_confidence_output' を明示する必要があるため。
diff レベルと session レベルで値域が異なる構造になる。

### 論点3.I: IG Score と target_score の責務分離

I1（統合）/ I2（維持）で議論。

→ Daiki 判定: I2

理由:
- IG Score は単独で「記事の構造的不足」を分析する価値あり
- 統合すると「対象選定スコア = IG Score の集計」という固定的な関係になり、IG Score の独立価値が消失

### 反証

**「3.H で H2（単一サービス）の方が YAGNI 原則に合う」**:
- Daiki 判定で H1 採用
- archive_service カラム追加と Adapter 層の Phase 1 工数増（推定 1-2日）はあるが、将来の切替時に「データ移行」と「実装変更」の2重負荷を回避
- 真=美の閉合性: 外部依存の抽象化を最初から組み込むことで構造的純度向上

**「論点3 で発見された問題が3件のみは過小では」**:
- 全17テーブル × 6カテゴリ = 102 検証ポイントを実施
- 問題3件は既に各 Step / 案で真=美テストを通過してきた累積結果
- 累積精度の証拠

### 重要な発見8: 4.M（同一記事の複数軸選定）の破綻リスク


### 解決策: M1' (queue ↔ session の多対多化)

新規テーブル master_rewrite_queue_session_link を導入:
```
master_rewrite_queue (軸別独立、複数選定理由を表現)
  ↕ master_rewrite_queue_session_link (多対多)
master_rewrite_session (1記事1セッション、運用感を維持)
```

体感複雑性:
```
M1素朴: 3倍（破綻）
M1':    1.05倍（中間テーブル1つ追加のみ）
M2:     1倍（だが軸履歴を捨てる）
```

→ M1' は M2 とほぼ同じ運用感でありながら、軸別選定理由を全て保持できる。

### γ2 の意味の再解釈

```
γ2 採用時の確定: queue_id NOT NULL（全 session が queue 経由）
↓
M1' 採用後: 「queue_session_link に最低1件のリンクを持つ」が NOT NULL の意味

→ session.queue_id カラム削除、master_rewrite_queue_session_link 経由で接続
→ アプリケーションレベルで「最低1件のリンク」を保証

```
