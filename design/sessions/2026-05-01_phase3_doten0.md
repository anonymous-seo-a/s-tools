## 3. 4案の真=美 3軸テスト評価

```
            必然性 | 閉合性 | 最小性 | 総合
(a) 移管統合 ◎    | ◎     | ◎     | 採用
(b) monitor.db同居 ✗ | ✗   | △    | データ層分離違反
(c) 既存廃止  ✗    | ◎     | ✗     | 既存運用消失 + 92件 seed 価値破壊
(d) 並走     △    | ✗     | ✗     | UI 二重化 + Daiki 認知負荷
```

### 各案の詳細評価

**(a) 移管統合**
- 必然性: master_annotations / master_rules / master_completeness_checklist / master_audit_log は全て **リライトシステムの責務範囲内**（注釈/表現ルール/整備進捗/編集履歴）。必然
- 閉合性: rewrite.db に集約することで「リライト関連マスター = rewrite.db」の閉合が成立
- 最小性: 全 24 テーブルが rewrite.db に揃い、命名空間も統一

**(b) monitor.db 同居**
- 必然性: ✗ monitor.db に 観測 + リライト + マスター を混在 → 案E のデータ層分離原則違反
- 閉合性: ✗ monitor.db が肥大化、責務不純（時系列観測 + 要因分析 + マスター混在）
- 最小性: △ DB は1個になるが、システム全体の責務は不純化

**(c) 既存廃止**
- 必然性: ✗ 廃止すると Compliance Layer が空洞化、92件 seed 価値消失（Daiki + ゆかちゃんの過去工数の物理的成果物）
- 閉合性: ◎ 表面的にはクリーン
- 最小性: ✗ 機能再実装の工数 + データ再投入

**(d) 別 DB 並走（既存→ master_legacy_*）**
- "legacy" prefix は虚偽（実装は active）
- Daiki が CRUD UI を二系統使い分け → 認知負荷
- 命名空間が二重化、UI 統合困難

## 4. 確定案: (a) 移管統合

### 確定事項

```
1. 物理配置: rewrite.db に移管（monitor.db からは Phase 4 着手時に DROP）
2. スキーマ: 既存実装そのまま維持（破壊的変更を避ける、最小性優先）
3. 命名: 現状維持（master_completeness_checklist と master_hcu_checklist の
         名前類似は用途明示で許容）
4. master_rules ↔ master_regulation_event: 粒度違いの親子関係として両立
   - 当面は master_rules.legal_basis (TEXT) で間接接続
   - Phase 3 後半で regulation_event_id 列追加を検討（任意）
5. master_audit_log: 汎用監査ログとして昇格、リライト関連マスター全体に適用
6. データ層分離原則の強化: monitor.db = 観測層 / rewrite.db = リライト層 + マスター層
```

### Phase 別配置の更新

```
[Phase E 既実装、Phase 4 で rewrite.db に移管: 4 テーブル]
  master_annotations / master_rules /
  master_completeness_checklist / master_audit_log
[Phase 1: 2 テーブル] 既設計どおり
[Phase 2: 12 テーブル] 既設計どおり
[Phase 3: 6 テーブル] 既設計どおり

合計: 20 → 24 テーブル
```

### Phase 4 実装時の物理移管手順

```
1. rewrite.db を新規作成（Phase 1 既設計どおり）
2. master-db.js の DB_PATH を rewrite.db に変更
3. monitor.db の Phase E 4テーブルを SQL DUMP
4. rewrite.db に CREATE + INSERT
5. 動作確認後、monitor.db からは 4テーブルを DROP
6. server.js の require パス（master-db / masters-routes）はそのまま
   （内部 DB_PATH のみ変更で透過的に切替）

※ Phase 4 着手時に Daiki に DROP 実行の最終承認を仰ぐ
```

## 6. 反証プロトコル

### 反証1: 「実は (b) が正しい?（DB 1個運用が真=美）」
better-sqlite3 は ATTACH DATABASE で複数 DB を同一トランザクションで扱える。
運用上のデメリットは小さく、責務分離のメリットが上回る。→ (a) を支持

### 反証2: 「master_rules + master_regulation_event は統合すべきでは?」
粒度が違う（法令イベント = 1件 / 運用ルール = N件、N対1関係）。
統合すると JSON 配列 or 重複行で表現することになり閉合性違反。
別テーブル + FK が真=美。→ (a) を支持

### 反証3: 「master_annotations は master_evidence で代替可能では?」
master_evidence: 一次情報（出典URL / アーカイブ）の永続化が目的。
master_annotations: 訴求語句 → 注釈テキスト + 注釈記号 のマッピング。
別概念、代替不可。→ (a) を支持

### 反証4: 「Phase E 4テーブルが Compliance Layer の実体だと、論点3 はほぼ既実装では?」
そのとおり。論点3 は「景表法/貸金業法/ASP規制の自動チェック」だが、既存 master_rules で
「禁止表現/必須表現/正式表記」の運用ルール集が既に整備されている。
論点3 の残作業は次の 4 点に縮小:
- LLM 出力の事後検証フロー（master_rules 参照ロジック）
- PR表記の存在検証（ステマ規制）
- 必須項目チェック（実質年率 / 限度額 / 返済方式 / 登録番号）
- master_regulation_event ↔ master_rules の親子接続（任意）
