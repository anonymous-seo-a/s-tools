## 2. 真=美 3軸テストによる構造分析

### 必然性テスト（タブの必須性）

```
既存 8タブ: 全て稼働中 → 必須

新規追加（必然性順）:
  リライトキュー   ★必須（工程1: 4軸独立キュー）
  リライト判定     ★必須（工程6: Daiki 真=美フィルタの実体）
  リライト履歴     ★必須（工程6'-7: セッション経過 + 反映状態）
  A/Bテスト        ▲ Phase 3 後半（Step A-4 実装時）→ 当面は履歴サブ画面で代用
  HCU結果          ✗ 独立タブ不要（差分判定画面のサブ表示で十分）
  関連記事         ✗ 独立タブ不要（差分判定画面 / 記事タブのサブ表示）
  Compliance結果   ✗ 独立タブ不要（マスタータブ + 差分判定画面で吸収）
```

### 閉合性テスト（命名空間衝突）

既存「承認」「履歴」「監査」と新規リライト系が同階層で並ぶと役割不明瞭。
解決: 系統別前置き接頭辞で分離。

```
既存「承認」 → 「CTA挿入」 （役割明示で改名、"承認" 単独だと何の承認か不明）
既存「履歴」 → 「CTA履歴」 （前置き接頭辞）
既存「監査」 → 「CTA監査」 （前置き接頭辞）
新規          → 「リライトキュー」「リライト判定」「リライト履歴」
```

### 最小性テスト（タブ数）

```
構成:
  [共通]    記事 / 順位モニタリング / マスター
  [CTA系]   CTA挿入 / CTA履歴 / CTA監査 / 商材 / ツール
  [リライト系] リライトキュー / リライト判定 / リライト履歴

合計: 3 + 5 + 3 = 11タブ
```

11 タブは多めだが、上位カテゴリ別の視覚区切り（CSS の border-left / spacing）で
認知負荷を抑制可能。階層化（ドロップダウン）はクリック数増 + Daiki の WMI 負荷
増のため避ける。

## 5. 反証プロトコル

### 反証1: 「11タブは多すぎ、階層化すべきでは？」
階層化（ドロップダウン）はクリック数増 + メニュー深さで Daiki の WMI 負荷を増やす。
フラット + カテゴリ区切りは1ステップで全画面到達可能、認知特性と整合。→ 維持

### 反証2: 「ヘッダタイトル変更は破壊的では？」
React テキスト変更のみ、運用影響なし。"s-tools" はリポジトリ名と一致、認知コスト低。→ 確定

### 反証3: 「既存タブ改名（"承認"→"CTA挿入"）は運用混乱では？」
ゆかちゃん含む運用者の慣れを破壊するリスク。だが「リライト判定」と並ぶと
どちらが Gap Fill か不明瞭になる方が問題。役割明示の利得が上回る。→ 改名確定

### 反証4: 「リライト判定にHCU/関連記事/Complianceを全部詰めると画面が重くなる？」
独立タブにすると Daiki がタブ間を行き来する負荷が増える。
リライト判定画面はサブパネル（折りたたみ可、デフォルト閉）で配置。
操作主軸は差分判定、補助情報はオンデマンド表示が真=美。→ サブパネル吸収確定

### 反証5: 「Phase 4 で実装してから決めるべきでは？」
ナビ構造はリライト判定画面のレイアウト前提（画面遷移、URL ルーティング、
ヘッダ高）に直結。設計フェーズで確定しないと 1-B のレイアウト議論が空転する。
→ 設計フェーズで確定
