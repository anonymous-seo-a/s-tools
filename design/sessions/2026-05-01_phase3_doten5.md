## 2. 5-1: Cloudflare Workers での variant injection

### Daiki 指摘への回答（更新頻度の整理）

「月次更新」の意味を分解:

```
(a) variant 割当の CF KV 更新頻度:
    テスト開始/終了の 2 イベントのみ (リアルタイム性不要)
    KV 伝播 (eventual consistency, ~1分) 許容

(b) 新しい A/B テスト開始のペース (SearchPilot 標準):
    テスト期間 6〜8 週間 + 準備 1〜2 週間 = 月次〜2ヶ月に1回
    3,000 記事規模なら Phase 4 初期は月次 1 サイクル
    Phase 4 安定期は並行 1〜3 テスト (change_category 8種で交絡なし)
```

### 実装方針

```
月次バッチ実行時 (新テスト開始時):
  1. master_ab_test に新テスト登録
  2. variant 割当 (control/A/B) を Cloudflare KV に push
     KV key: post_id, value: {variant, test_id, applied_at}

リクエスト時 (Cloudflare Worker):
  1. URL から post_id 抽出
  2. KV から variant 取得 (latency 数ms)
  3. WordPress オリジン HTML をフェッチ
  4. variant に応じて HTML 改変:
     - control: そのまま返す
     - A / B: variant 別 master_rewrite_diff.content_after を適用
  5. クライアントに返す
```

実装位置: `node/rewrite/ab-testing/cf-kv-pusher.js` (新規) + Cloudflare Workers スクリプト

### 反証

```
反証: 「KV ではなく WordPress プラグインで variant 切替する方が単純では？」
  → サーバー側切替だとキャッシュ層 (Cloudflare CDN) で control/variant が
    混在するリスク。Edge layer (Workers) での切替が SearchPilot 方式の標準。

反証: 「KV の 1分伝播で運用に支障は？」
  → A/B テストは月次セットアップ、リアルタイム性不要。
```

## 3. 5-2: 層別ランダム割付

```javascript
function stratifiedAssign(posts) {
  const result = { control: [], variantA: [], variantB: [] };
  const byCategory = groupBy(posts, 'category');
  for (const [category, list] of Object.entries(byCategory)) {
    shuffle(list);  // Fisher-Yates
    const n = list.length;
    result.control.push(...list.slice(0, n / 3));
    result.variantA.push(...list.slice(n / 3, 2 * n / 3));
    result.variantB.push(...list.slice(2 * n / 3));
  }
  return result;
}
```

理由:
```
- カテゴリ別の交絡排除（cardloan/crypto/securities/fx 各 1/3 ずつ）
- SearchPilot 方式の標準アプローチ
- 統計的検出力: 各群 ~800 で成立、1,000 は余裕
```

実装位置: `node/rewrite/ab-testing/variant-assigner.js` (新規)

## 4. 5-3: 観測期間 + 統計検定（調整可能性確保）

### Daiki 指摘の反映

「いったん OK、後で結果みて調整できる余白を残したい」
→ スキーマレベルで余白を確保（Phase 4 後の拡張で ALTER 不要）

### スキーマ拡張: master_ab_test +statistical_method

```sql
ALTER TABLE master_ab_test ADD COLUMN statistical_method TEXT DEFAULT 'bayesian_final';
-- 取りうる値:
--   'bayesian_final'      終了時 1 回の Bayesian credible interval (MVP)
--   'bayesian_sequential' 週次チェック + early stopping (Phase 4 後拡張)
--   'frequentist_t_test'  頻度主義 t 検定 (代替)
--   'custom'              個別 hypothesis に応じたカスタム
```

### デフォルトパラメータ（MVP）

```
applied_at:                 variant 適用日 (CF KV push 完了時)
observation_start:          applied_at + 7日 (indexing 安定化)
observation_end:            applied_at + 56日 (8週間)
statistical_method:         'bayesian_final'

調整可能パラメータ:
  observation_start/end: 既存 DATETIME 列、個別テストで上書き可
  statistical_method:    新規列、運用結果見て変更可能
```

### 警戒バイアス対チェック

```
- 「機能を盛りたくなる」: statistical_method 1 列追加は minimal hook、
  「余白を残す」と「過剰精緻化」のバランス。許容範囲。
- 「過剰精緻化」: early stopping は MVP 対象外、デフォルト固定で開始。
```

## 5. 5-4: Google Update 期間中の信頼度補正

```
Update 検出:
  monitor-jobs.js に検出 cron 追加
  Google Search Status Dashboard / RSS 監視 (Adapter パターン)
  Update 期間を SQLite に記録

master_ab_test_result.google_update_in_period (既設計、BOOLEAN):
  観測期間中に Update があった場合は true に更新

補正方式:
  完全重複 (観測期間 ⊆ Update 期間):
    結果破棄、master_ab_test.status='aborted' で再テスト
  部分重複 (観測期間 ∩ Update 期間 ≠ 空):
    Update 期間を除外して再計算、master_ab_test_result.notes に除外期間記録
  重複なし:
    通常通り
```

実装位置: `shared/google-update-monitor.js` (新規、Adapter パターン)

## 8. 反証プロトコル

### 反証1: 「もっと短いサイクル (週次) で回転させた方が学習速度が上がるのでは？」
短サイクルは indexing/ranking 安定化前にテスト終了 → 統計的に無効。
Google の確認済み: ranking 安定化に最低 4〜6 週間必要。
SearchPilot 標準の 8 週間が真=美。

### 反証2: 「並行 3 テストは交絡リスクが高い？」
change_category が異なれば交絡しない (例: title 変更 と h2構造変更 は独立)。
同一 change_category 内では 1 テストのみ。

### 反証3: 「early stopping を入れた方が効率的では？」
MVP 対象外（handoff 警戒バイアス「機能を盛りたくなる」）。
statistical_method 列で将来拡張可能、運用結果見て判断。

### 反証4: 「Status Dashboard スクレイピングは脆弱では？」
RSS フィード優先。Adapter パターンで将来差し替え可能（案D 3.H 同パターン）。
