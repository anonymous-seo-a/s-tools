## 論点3: 採用却下の構造的理由

### # 7 階層的 schema（不採用）

**却下理由**: CMS（WordPress）側で扱う領域

```
構造的位置:
  記事リライトの差分パッチではなく、CMS テーマ・プラグイン層で出力すべき構造化データ

真=美評価:
  必然性: 強（公式 docs 直接支持）
  閉合性: × リライトシステムに含めると責任境界が混濁
  最小性: リライトシステム外で対応すれば最小性に整合

警戒バイアス排除:
  「強推奨ラベルへの追従バイアス」を回避
  一次情報支持があっても、責務分離の観点で却下する判断は構造的に正当
```

### # 10 Core Web Vitals (INP) 監視（不採用）

**却下理由**: リライト主流路外 + Clarity との機能重複

```
構造的位置:
  記事本文・schema のリライトでは INP は変化しない（JS・テーマ層の問題）

真=美評価:
  必然性: × リライト効果測定の必須要素ではない
  閉合性: × Clarity engagement_score と機能重複
  最小性: 永遠に外側で最小性整合

Clarity との重複の具体:
  Clarity の dead_click_rate / rage_click_rate / engagement_score = ユーザーの実体験
  INP = 技術的応答性
  両者は近接概念、リライトシステムでは Clarity 側で十分
```

### Daiki の判定（論点3）

「却下理由の明文化に追加・修正・反証はない」 → 確定

## 採用却下した選択肢の詳細理由

### 階層的 schema（# 7）リライトシステム組込み
- CMS（WordPress）側で扱う領域
- 一次情報支持があっても責務分離が構造的に正しい

### INP 監視（# 10）リライトシステム組込み
- リライト主流路外（記事本文リライトで INP は変化しない）
- Clarity engagement_score と機能重複
- 既存 monitor 系で扱う領域

### Author expertise schema（# 6）リライトシステム側で構造保存
- CMS 側で構造化
- 監修判断は人間
- 今回フェーズ対象外

### Reddit / Quora API 連携（# 4 拡張）
- Reddit API 課金化（2023年4月以降）
- Quora 公式 API なし、scraping 規約違反リスク
- Yahoo知恵袋 規約上スクレイピング禁止
- → PAA + サジェスト + 関連検索のみで Step A-2 既存スキーマで完結

### Content decay 軸4 を軸2 サブシグナルに統合
- 軸2 = 「現時点の impressions × position_gap_factor」（ポテンシャル指標）
- 軸4 = 「実測 traffic 28日 vs 90日 差分」（実測劣化シグナル）
- 異なる事象を捉えている → 独立キューが真=美に整合

### # 9 レベル0/1/2

### 1. 強推奨ラベルへの追従バイアス（新規発見）

```
リサーチレポートの「強推奨」評価は、出典の質に幅がある:
  - Google 一次情報 + 業界専門家複数支持: 強い根拠
  - 業界専門家広範支持: 中程度の根拠
  - ベンダー単独支持: 弱い根拠

→ ラベルだけで採用すると、根拠の弱い手法も組み込まれる
→ 各手法を Google 一次情報の有無で再フィルタリングが必須

実例: # 9 Hub-and-spoke は Google 一次情報の直接支持なし
       # 1 Content decay も同様に間接的（HCU との整合性のみ）
       # 2 Cannibalization も同様
```


### 3. システム全体の最小性 vs テーブル単位の最小性

```
真=美の最小性は、テーブル単位ではなくシステム全体で評価すべき場合がある

実例: master_article_similarity
  単独で見ると # 9 の他に必要ないように見える（テーブル単位では過剰）

→ 設計判断時、「他用途流用可能性」を必ず検討する
```

### 4. 責務分離原則の拡張適用

```

```
