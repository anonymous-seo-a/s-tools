## Phase 0: リサーチお題設計

### 議論の出発点

セッション開始時、Daikiから「日米独英のSEO手法を網羅的にまとめ、AI駆動リライトシステムの設計に活用したい」との要請。

私の初期反応（Web検索ベース）:
- 米国: コンテンツ駆動型（Skyscraper Technique等）
- ドイツ: テクニカル駆動型（Sistrix Sichtbarkeitsindex等）
- 英国: 規制駆動型（FCA + E-E-A-T）
- 日本: 順位駆動型（リライトサイクル）

### 重要な軌道修正1: 工程主軸への変更

私が当初「国別軸」で整理しようとしたが、Daiki から以下の指摘：
> 「工程別軸の方が AI リライトシステム設計に直結する」

私の自己検証:
- 必然性テスト: 国別軸はリライトAI設計に必須か → **不要**
- 閉合性テスト: 国別深掘りは「なぜこの手法を採用したか」を閉じるか → **閉じない**
- 最小性テスト: 20項目は最小か → **過剰**

→ 工程主軸への再構成。ただし国別差分は「Phase 2の各工程内で言及」する形で保持。

### 重要な軌道修正2: ポジトーク回避の運用ルール確立

私が当初「実務家が使っている手法」を有効性根拠としていたが、Daiki から以下の指摘：
> 「SEOに関するweb上の情報のほとんどはポジトークだ。明確にスパムではなく、かつ、Googleの検索セントラルの情報から、加点要素があると判断できる内容を『現在有効』と判断して良い」

これにより、出典評価の3層ヒエラルキーを確立：

```
強い出典: Google公式（Search Central, Quality Rater Guidelines等）
中程度: 複数独立ソースで交差確認できる情報
弱い出典: 個別SEO会社単独ブログ、自社ツール宣伝記事
  → 弱い出典のみで支持される手法は採用判断を保留
```

### 重要な軌道修正3: 「日本未採用×海外採用×Google支持」の最優先化

Daiki の元々の意図は「日本で実施されていないが効果がある海外の手法を取り入れる」こと。
私が工程主軸に変更したことで、この核心が薄れていた。

修正: Phase 2 の各工程で、以下の3カテゴリ分類を必須化：
- (a) 日本で広く採用されている標準手法
- (b) 日本で未採用/低採用だが、海外で採用されている手法 ← 最優先
- (c) Google一次情報で支持されているが、日本で見落とされがちな手法

リサーチ終了条件として「(b) カテゴリの手法が最低5項目抽出されている」を追加。

### Phase 0 確定したリサーチ・プロンプト

最終的に Researchモード用プロンプトとして確定。主要構成：
- Phase 1: 環境確認（HCU以降のSearch Quality Rater Guidelines改訂）

## Phase 2: Step A-1 (Information Gain 駆動の差分生成)

### 議論の出発点

Information Gain は工程5の核。Google特許とHCU公式質問の両方で支持されており、出典の質は「強」。

私の自己検証:
- Information Gain は Daiki の「真=美」と構造的に同型
- Information Gain高い記事 = 構造的に閉じている = 真=美の閉合性を満たす

### 設計判断ポイント1: 「事実集合」の定義

「事実」をどう機械的に扱うかの核心判断。3つの候補レイヤー：

- **レイヤー1（エンティティ単位）**: `金融庁` `貸金業法` `総量規制` 等
- **レイヤー2（事実主張単位）**: `プロミスの上限金利は17.8%` 等
- **レイヤー3（経験単位）**: 自社の一次体験（フォーム所要時間実測等）

私の推奨: 3レイヤー全採用、ただし役割を分離

各レイヤーの強み・弱み:
- レイヤー1: 機械抽出が確定的、ただし「事実」ではなく「単語」
- レイヤー2: 事実を直接扱える、ただし hallucination リスク
- レイヤー3: 競合との差分が構造的に保証される、ただし機械化困難

**Daiki の判定: 案1（3レイヤー全採用）**

### 設計判断ポイント2: KPIの選定

候補:
- 案1: Information Gain スコア（特許準拠、ただしGoogle実装未確認）
- 案2: HCU質問項目充足度（公式質問準拠、より安全）
- 案3: 両方併記

私の反証: Google は「特許 ≠ 実装の確証」と公式に言及。HCU質問項目の方が安全。

**Daiki の判定: 案1（Information Gainスコア直接採用）**

判断の含意:
- Google実装の有無は問わない
- 真=美と構造同型である以上、自社の品質判定指標として直接使う
- HCU 22項目チェックリストは別工程（工程5-c Topical depth scoring）で別KPIとして測定される

### 設計判断ポイント3: 新規テーブル分離

Daiki: 「合意、よくわかってないけど、この辺はあなたに任せた方が最適でしょ」

確定した新規テーブル:
- master_fact_set
- master_competitor_corpus
- master_information_gain_score

### 設計判断ポイント4: 競合上位URL数

候補: N=3 / N=5 / N=10 / 動的

レポート全体を通じて「上位10サイト分析」が日本標準として批判されていた。N=3は意図的な絞り込み。

**Daiki の判定: N=3、ただし設定で可変**
- SERP intent shift 検出時に自動引き上げ可能な構造

### 反証

**「Information Gain をシステムの核にして良いか」**:
- Google公式は特許≠実装と言及
- ただしHCU公式質問 "Does the content provide insightful analysis or interesting information that is beyond the obvious?" は Information Gain の概念そのもの
- → 採用するが、絶対値ではなく相対比較で運用するのが安全

**「3レイヤー全採用は重すぎないか」**:
- レイヤー2の実装コスト高
- ただしCompliance Layer との連携で必須
- → 実装順序として レイヤー1 → レイヤー3 → レイヤー2 が現実的

## Phase 2: Step A-2 (Query Fan-out 構造シミュレーション)

### 議論の出発点

Query Fan-out は工程2の核。Google公式blog（developers.google.com/search/docs/appearance/ai-features）と特許で直接支持。出典の質は「強」。

### 設計判断ポイント1: Fan-out の生成主体

候補:
- 案1: LLM単独で生成（低コスト、Google実装との乖離リスク）
- 案2: 実SERP起点で逆算（Google実装に近い、既存枠を超えない）
- 案3: ハイブリッド（LLM生成 + 実SERPで検証・補強）

私の推奨: 案3
- LLM単独はGoogle実装との乖離リスク
- 実SERP単独は既存の枠を超えない（「安心ニーズ」が拾えない）
- ハイブリッドは Google一次情報を**根拠**として使い、LLMを**展開器**として使う

**Daiki の判定: 案3（ハイブリッド）**

### 設計判断ポイント2: 分解次元

工程3（Micro-intent分解）との接続をどうするか：
- 案1: Fan-out の出力をMicro-intent 5次元にマッピング
- 案2: Fan-out独自の次元（主題ベース）
- 案3: 二段構造（Layer1主題分解 + Layer2 micro-intent展開）

私の推奨: 案3
- Query Fan-out は Google の挙動の模倣（主題分解）
- Micro-intent は検索者の認知構造の分解
- 両者は別レイヤーで、組み合わせると網羅性が最大化

**Daiki の判定: 案3、ただし Layer2 への競合URL収集は行わない**
- Layer2はセクション設計の参考情報として使う
- コスト効率の観点から合理的判断

### 設計判断ポイント3: AI Overview citation源

候補:
- 案1: 自然順位のみ追跡
- 案2: AI Overview citation も別途追跡
- 案3: 統合追跡（source_type カラムで区別）

**Daiki の判定: 案3（統合追跡）**
- 初期は「AI Overview の有無の記録のみ」、citation源URL取得は後追加可能

### 設計判断ポイント4: 工程2と工程3の統合

私の提案: 工程2-3 統合（Query Fan-out が両方を担う）

**Daiki の合意**

### 設計判断ポイント5: SERP API 選定（最も議論が長かった部分）

#### 第1回提案: DataForSEO 第一選択
- 月3,000-5,000クエリ規模で月$10-20の低コスト
- SEO業界の事実上の標準

**Daiki の判定**: 速度遅すぎ（45分待ち）でNG
- 「速度かつ精度を求める。費用はそこまでかけられない」

#### 第2回提案: Scrape.do 第一選択
- 1.73秒（最速）、AI Overview検出60%、PAA完全取得
- 月$3.50-5.80（最安級）

**Daiki の判定**: SerpApi のサブスクで決定
- 精度最高（AI Overview検出68%、PAA完全）
- Developerプラン $75/月

#### 学び

私の選定プロセスの誤り:
1. 当初DataForSEOを推した時に、応答速度の確認が甘かった
2. 「実コンテンツ取得」工程を見落としていた
3. 「精度」の解釈軸を初期に明確化していなかった

軌道修正後の最終形:
- SerpApi（Starter $25/月）から開始
- 抽象化レイヤー（Adapter層）を挟んで将来差し替え可能に

#### 重要な軌道修正4: Yahoo!スクレイピングの除外

Daikiから「APIは品質低いからYahooのスクレイピングで代替できると良いのだけど」との要請。

調査結果:
- Yahoo! JAPAN利用規約で明示的に禁止
- Yahoo! JAPANはGoogleの検索エンジンを使用 → Yahoo!を取得してもGoogle代替にならない
- 法的リスク + 技術的リスク（アクセス遮断）

→ Yahoo!スクレイピング除外、SerpApi採用へ

### 反証

**「LLMによる Layer1 主題分解は、Google の実際の fan-out と乖離する」**:
- 完全一致は不可能、ただし実SERP の PAA・関連検索を参考情報として渡すことで近似可能
- generation_method で識別、後で検証可能

## Phase 2: Step A-3 (Evidence Layer)

### 議論の出発点

最優先5手法のうち2つ（Original research, First-hand experience markers）を**統合した Evidence Layer** として設計。

私の自己検証:
- 両者とも「自社が一次的に取得した、競合が持ち得ない情報」
- 共通構造として一元管理が真=美の最小性に合致

### 設計判断ポイント1: Evidence の分類軸

候補:
- 案1: 形式分類（数値/画像/動画/音声/文字記録）
- 案2: 生成方法分類（アンケート/実測/ヒアリング/一次体験記録/公式データ照合）
- 案3: 用途分類（IG用/E-E-A-T用/Compliance用/信頼補強用）
- 案4: ハイブリッド（形式 × 生成方法 × 用途の3軸タグ）

私の推奨: 案4
- 1つのEvidenceは複数用途で使われることが多い
- タグで管理することで、用途別の検索・引用が可能

**Daiki の判定: 案4（ハイブリッド）**

ただし最終的には4軸タグ（形式 × 生成方法 × 用途 × 変動性）に拡張。

### 設計判断ポイント2: Evidence と記事の関係

候補: 1対1 / 多対多 / テーマ別共有

私の推奨: 多対多
- 同じ Evidence は複数記事で引用される
- 1対1だと取得コストが膨大

**Daiki の合意**

### 設計判断ポイント3: 劣化管理

金利・限度額・融資時間は変動する。

私の推奨: volatility（変動性タグ） + valid_until（有効期限） 併用
- 金利・限度額: high volatility → 短い valid_until（1ヶ月）
- 法令引用: static → valid_until 不要
- 一次体験スクリーンショット: low volatility → 長い valid_until（6ヶ月）

**Daiki の合意**

### 設計判断ポイント4: URL検証

私の推奨: Wayback Machine自動アーカイブ + 月次健全性チェック
- 公式情報源（金融庁、消費者庁、各社公式）は定期的にURL変更がある
- スナップショットで「取得時点の内容」を保証

**Daiki の合意**

### 確定した新規テーブル

- master_evidence
- master_evidence_article_link

加えて、master_fact_set にevidence_idカラムを追加して接続。

### 反証

**「Evidence Layer は人間が手動運用するため、システム化の効果が薄い」**:
- 取得は人間でも、運用（劣化管理・URL検証・引用照合）はシステムで自動化可能
- 自動化されないと、500件のEvidenceを人間が管理するのは破綻
- → システム化の必然性は高い

## Phase 2: Step A-4 (SearchPilot方式 SEO A/B Testing)

### 議論の出発点

最優先5手法の最後の1つ。工程8（効果測定・学習ループ）の核。

### 重要な前提変更（議論の途中で判明）

**当初の前提**:
- 月50記事を Daiki が手動で回す（Daiki の判断容量上限）
- 記事数が少ない（50）→ 完全SearchPilot方式は統計的検出力不足
- WordPress + Xserver で CDN なし

**Daiki から判明した実際の前提**:
- 既存3,000記事すべてが対象
- 月50記事はあくまで Daiki の判断容量の上限
- CDN導入済み（Cloudflare Workers利用可能）

これによりStep A-4 全体の設計が現実離れしていたことが判明。

### 設計判断の再構築

#### 設計判断ポイント1: A/Bテスト方式

旧推奨（記事数50想定）:
- 案4（Before/After比較）必須採用
- 案1（完全SearchPilot方式）は記事数200以上で再検討

新推奨（記事数3,000想定）:
- **案1（完全SearchPilot方式）採用**
- Cloudflare Workers でvariant injection
- 1,000記事ずつ control / variant A / variant B に分割

**Daiki の判定: 案1採用**

#### 設計判断ポイント2: 学習ループ

私の初期案: パターン抽出（master_rules反映）

Daiki の指摘: **「再設計の必要がありそう」**

私の自己検証:
- 3,000記事のテスト結果を人間が一件ずつ判定する設計は破綻する
- パターン検出を自動化する必要がある

新案: **3層学習ループ**
- Layer1: パターン記録（記述的、自動分類）
- Layer2: パターン抽出（推論的、100テスト累積で統計的有意な抽出）
- Layer3: 戦略反映（処方的、master_rules 自動反映 + Daiki承認）

**Daiki の合意**

#### 設計判断ポイント3: 効果測定指標

主要4指標: クリック / インプレッション / CTR / CV

Daiki の指摘: **「可能なら microsoft clarity の情報も活用できると理想」**

#### 設計判断ポイント4: Microsoft Clarity の位置づけ（再議論）

私の初期案: 「リライト計画時の補助指標、A/Bテスト判定には使わない」

Daiki の指摘: **「A/Bテスト判定にも活用」**

新案: **主指標 × Clarity gating の二段判定**
- 主指標群（クリック/インプ/CTR/CV）の統計的有意差
- AND
- Clarity指標群が悪化していないこと（gating condition）

具体例: クリック+10%、CV+5%、しかしレイジクリック率+30% → `positive_with_warning` として記録

**Daiki の合意**

#### 設計判断ポイント5: SerpApi プラン要件

私の初期計算（記事数3,000フルスキャン想定）:
- 初回フルスキャン Production プラン（$150/月、1ヶ月のみ）
- 継続運用 Developer プラン（$75/月）

Daiki の指摘: **「初回で3000記事全てスキャンする必要はあるのか？リライトプログラムは実運用しながら少しずつ修正&拡張していくから、細かく動かせるようにしておきたい」**

私の自己検証:
- 「初回フルスキャン」概念は真=美の最小性テストに反する
- 月50記事ペースなら、月50記事分のスキャンで足りる

## 採用却下した選択肢の詳細理由

### Yahoo!スクレイピング
- **理由**: Yahoo! JAPAN利用規約で明示禁止

- **法的リスク**: 利用規約違反 → 民事上の責任の可能性

### DataForSEO（標準速度）

  - PAA回答全文取得不可（質問のみ）

### Scrape.do
- 速度1.73秒（最速）、AI Overview検出60%、PAA完全取得
- 価格$1.16/1,000で最安級

## 学び（このセッションを通じて確立されたもの）

### 1. 真=美フィルタの実適用パターン

各設計判断で機械的に適用：
- 必然性テスト: この要素を除いたら構造は崩壊するか
- 閉合性テスト: 内部で全ての関係が完結しているか
- 最小性テスト: より少ない要素で同じ機能を実現できないか

具体的適用例:
- 工程主軸への変更（国別軸の必然性が無い）
- 遅延評価への変更（初回フルスキャンの最小性違反）
- 3レイヤー Information Gain（各レイヤーに独自役割）

### 2. ポジトーク回避の運用ルール

```
強い出典: Google公式（Search Central, QRG, blog.google）
中程度: 複数独立ソースで交差確認
弱い出典: 個別ベンダー単独支持
  → 弱い出典のみで支持される手法は採用判断を保留
```

業界通説の多くは弱い出典しか持たない。これらをシステムKPIに採用しないことが、長期的な競合優位の源泉となる。

### 3. AI vs 人間の役割分担

| 役割 | 担当 |
|---|---|
| データ収集・構造化・計算 | AI |
| 真=美の最終判定 | 人間（Daiki） |
| USPの言語化 | 人間（Daiki / Arai） |
| 著者・監修者の本物性 | 人間 |
| 規制の最終解釈 | 人間 |

LLMは「自由生成器」ではなく「制約付き構造変換器」として位置づける。

### 4. 反証プロトコルの定着

各設計判断で必ず反証を提示。Daikiの直感は片側照射の傾向があるため、AIが補完として逆方向検証を提示する。
このセッションでも、反証を通じて以下が修正された：
- 工程主軸への変更（私の片側照射）
- ポジトーク回避（私の見落とし）
- Microsoft Clarity のA/Bテスト統合（私の限定的解釈）

### 5. 前提変更への対応速度

Daiki が前提を更新（例: 50記事→3,000記事）すると、設計の根幹が変わる。
私の対応として、自己検証パターンを確立：
- 旧前提の何が変わるか
- 旧設計の何が現実離れしていたか
- 新前提でどう設計し直すか

このパターンを Step A-4 で実行。記事数前提変更により、Step A-4 の設計が大幅に再構築された。

### 6. 段階的拡張（MVP→ML）の設計

学習ループ Layer2（パターン抽出）は初期は単純統計、累積後に機械学習統合検討。
Information Gain は初期はレイヤー1+3、後にレイヤー2追加。
SerpApi は初期Starter、必要に応じて拡張。

これらは「最小性」と「拡張可能性」を両立する設計パターン。

### 7. 抽象化レイヤーによる将来差し替え可能性

API Adapter層を挟むことで、SerpApi → Serper / Scrape.do への将来切り替えが容易に。
この設計パターンは Information Gain の3レイヤー、A/Bテストの3層学習ループにも応用可能。
