## 論点1: buildPrompt の目的一致確認と本文取得方法

### 議論の出発点

ハンドオフプロンプトで警戒指定された「既存資産への過剰適応バイアス」を最初に潰す必要があった。

monitor-analysis.js の buildPrompt は要因分析用に高品質な実装。これを案C で全面流用する誘惑があったが、**目的が違う**ことを最初に確認。

### 概念レベルの目的差分

| 軸 | C系: 要因分析生成 | 案C: リライト指示書生成 |
|---|---|---|
| 目的 | 「なぜ指標が変化したか」を説明 | 「どう変更すべきか」を提案 |
| 出力形式 | 自然言語コメント | 構造化された変更（差分 or 指示） |
| 検証可能性 | 低（解釈依存） | 高（A/Bテスト対象） |
| 副次効果 | なし（観測のみ） | 大（記事を実際に書き換える） |
| 失敗コスト | 低（誤った解釈の蓄積） | 高（誤った差分が公開される） |

→ **目的は別**。案EのA系/B系と同型の構造的差異。

### 3層分解の確立

```
Layer 1: データ集約（DB→構造化データ）
  → 流用可能（目的非依存）

Layer 2: コンテキスト整形（構造化データ→プロンプト文字列）
  → 部分流用 + 拡張（不足項目あり）

Layer 3: タスク指示（プロンプト末尾の「何をさせるか」）
  → 完全新規（目的が違う）
```

### Daiki の判定（論点1-A/D）

- 論点1-A（Layer 1 新規追加データ8項目）: AI に一任
- 論点1-D（関数構造）: B採用（buildRewritePrompt を新設、共通ヘルパー切り出し）

### 重要な軌道修正1: 本文サイズ実態の判明

Daiki の指摘「記事本文のキャップは必要か？多分13000文字だと収まらない」が決定的だった。

私の初期試算（記事最大40,000文字想定）では「200K context に余裕で収まる」と楽観視していたが、実態調査で：

```
実測値（n=100、母集団 2,294記事）:
- min: 26,275 文字（既存キャップ12,000の2倍超）
- p50: 114,178 文字
- p95: 165,986 文字
- max: 210,141 文字

重大な発見:
- 既存 MAX_CONTENT_CHARS = 12,000 は全記事で発火
- 既存システムは記事の冒頭5-10%しか LLM に見せていない状態
```

→ 前提崩壊。単純なキャップ拡張では解決不可能。

### 重要な軌道修正2: 構造化抽出による削減

Daiki の指摘「style タグなども含めた文字数を抽出している。純粋な見出しとテキストだけで判定できるようにすれば改善するはず」を起点に、構造化抽出版を試作。

実測結果（n=3）:
- 18755: 210K → 37K（82.5%削減）
- 8987: 154K → 27K（82.5%削減）
- 29802: 181K → 33K（81.9%削減）

→ **案E（構造化抽出）採用、案F（ハイブリッド）不要**。

3記事すべてで削減率82%が一致 = 構造的傾向（個別事情ではない）。

### 学び: 真=美の最小性が機能した瞬間

```
誤った道:
  本文サイズ問題 → キャップ拡張で対応 → 階層化処理（案D）
  → 複雑な実装、コスト増

正しい道:
  本文サイズ問題 → 根本原因（タグ間空白の累積）を除去 → 構造化抽出
  → シンプル、コスト不変
```

「機能を足す」のではなく「ノイズを削る」ことが本質だった。

### 追加削減フィルタ

Daiki: 「全採用」

確定:
- 監修者ブロック除去（全記事共通テキスト）
- 目次（ez-toc）除去（h タグ階層で代替）
- CTA バナー → [CTA: {partner_slug}] マーカー化（位置情報を保持）

→ 推定さらに 20% 圧縮（max級記事 38K → 25-30K）。

### 反証

**「構造化抽出は本当に削減になるのか」**:
- Markdown形式の見出しマーカーや表区切りは増加方向にも働く
- ただし実測 82% 削減 = 構造的傾向で確定

**「3記事サンプルで判断するのは早計」**:
- 削減率の一致（82.5/82.5/81.9）は構造的傾向の証拠
- 個別記事のバラつきは限定的
- → n=3 で十分な判定可能

## 論点2: 出力形式（案H 差分パッチ採用）

### 議論の出発点

LLM から何を出させるか。設計の根幹を左右する判断。

### 4案の評価

| 案 | 出力形式 | 概要 |
|---|---|---|
| 案G | 完全書き直し | 記事全文を新規生成 |
| 案H | 差分パッチ | 変更箇所のみセクション単位で出力 |
| 案I | リライト指示書 | 自然言語の指示書、実装は別工程 |
| 案J | ハイブリッド | 指示書 + 差分の二段階 |

### マトリクス評価

| 軸 | 案G | 案H | 案I | 案J |
|---|---|---|---|---|
| トークン効率 | × | ◎ | ◎ | △ |
| 真=美判定 | △ | ◎ | ◎ | ◎ |
| 工程7接続 | ◎ | ○ | × | ○ |
| 失敗時安全性 | × | ◎ | ◎ | ◎ |
| A/Bテスト接続 | △ | ◎ | △ | ◎ |
| 実装難度 | ○ | △ | ○ | × |

### Daiki の判定

「案Hで案Jへの拡張を後ほど検討」

### 案G を退ける根拠

- 全文書き直しは既存記事の蓄積価値（既に検索順位を持っている既存構造）を破壊
- A/Bテストの粒度が荒く、Step A-4 の3層学習ループが機能しない
- Daiki の真=美判定の対象が「全文比較」になり、認知リソース消費が膨大

### 案I を退ける根拠

- 工程7（実装）への接続が弱い
- 「指示書 → 実装」の翻訳コストが LLM 二段階 or 人間ライターを要求
- 自走システムとしての完結性が低い

### 差分構造の確定

```typescript
type RewriteDiff = {
  diff_id: string;
  target_section: string;
  change_type: ChangeType;  // 9種類
  rationale: { ... };
  content: { before, after };
  estimated_impact: { change_category, confidence };
};
```

change_type 9種類:
- rewrite_section / rewrite_paragraph
- insert_after / insert_before / insert_evidence
- delete_section
- update_title / update_meta_description
- restructure_outline

### 反証

**「案H は全文書き直しが必要なケースに対応できない」**:
- 全文書き直しが必要な記事は対象選定で別ルートに分岐
- 軸1スコア極端に低い記事は「リライト不可、新規記事化」として運用ルールで除外

**「change_type 9種類は最小性に反する」**:
- Step A-4 change_category と1対1対応 → 統合すると追加マッピング層が必要
- 現状が最小

## 論点3: LLM 選定（役割別モデル分離）

### 議論の出発点

候補モデル: Anthropic Claude Opus 4.7 / Opus 4.6 / Sonnet 4.6 / Haiku 4.5、OpenAI GPT-5.5 / GPT-5.4。

### 私の初期推奨（変更前）

「Phase 1 主候補 Sonnet 4.6、副候補 Opus 4.7」

理由: コスト効率（月$20）、SWE-bench差わずか1.2pt。

### Daiki の判定（論点3-1）

「分析や構造判断はOpusの最新モデル、コンテキストが多くなるライティングはsonnetの最新モデル」

→ **役割別モデル分離戦略**が確立。

### 構造的含意

これは案J（ハイブリッド: 指示書 + 差分）の前倒し発生。

```
工程6'-A: 分析・構造判断 → Opus 4.7（推論重視）
工程6'-B: 差分テキスト生成 → Sonnet 4.6（ライティング重視）
```

設計上の妥当性:
- 真=美の必然性: 分析と生成は本質的に別タスク
- 真=美の閉合性: モデルの強みを各タスクに合致
- コスト効率: Opus 全工程運用（$33.5）と同コストで品質向上

### 重要な軌道修正3: コスト試算の更新

```
工程6'-A (Opus 4.7):
  入力 80K × $5/M + 出力 2.5K × $25/M = $0.46

工程6'-B (Sonnet 4.6):
  入力 18K × $3/M + 出力 10K × $15/M = $0.21

計 $0.67/記事 × 50 = $33.5/月
```

→ Opus単独運用と同コストで品質向上。役割分離は構造的に純利。

### Daiki の判定（論点3-2/3/4）

- 論点3-2（モデル選定戦略）: 「アウトプットを見てどちらにするか選びたい。トグルボタンなどで選択しながらテスト」
- 論点3-3（Adapter層）: 「将来的に画像生成が必要になるかも。OpenAI APIとGemini APIにも一応対応」
- 論点3-4（キャッシング）: 「OK」（Phase 4 で実装）

### Adapter 層の拡張版設計

```
Phase 1 から3社対応:
  - AnthropicAdapter
  - OpenAIAdapter（将来: dall-e-3, gpt-image-1）
  - GeminiAdapter（将来: imagen-3）

UI:
  - トグルボタンでモデル動的切替
  - A/B 比較実行モード（2モデル同時実行）
```

### GPT 系を退ける根拠

- GPT-5.5: ハルシネーション弱点 → YMYL 致命的
- GPT-5.4: 日本語品質・推論精度で Claude 劣勢、切替理由弱い

ただし Adapter 層は3社対応で設計（Daiki 指示）。

### 反証

**「役割別モデル分離は本当に必要か」**:
- ベンチマーク差は小さいがタスク特性差は実在
- トグルボタンで動的切替可能
- → 段階的検証で対応可能

**「3社 Adapter は YAGNI 違反では」**:
- Daiki 指示「画像生成必要になるかも」
- 抽象化コストは小さい
- 後追加より最初から3社対応の方が責任境界明確

## 論点4: 人間判定境界（案L ベース + 案K 発動）

### 議論の出発点

LLM がどこまで実行し、Daiki がどこから判定するか。

### 4案の評価

| 案 | 境界1（方針判定） | 境界2（差分判定） | LLM呼出 | Daiki 認知負荷 |
|---|---|---|---|---|
| 案K | あり | あり | 2回 | 高 |
| 案L | なし | あり | 2回連続 | 中 |
| 案M | あり | なし | 1回 | 低 |
| 案N | なし | なし | 1回 | ゼロ |

### Daiki の判定

「推奨」（案L ベース + 高リスク時に案K 発動）

### 推奨理由

#### 1. Daiki 認知負荷の最小化

ナレッジファイル1の確認:
- WMI/PSI が PRI/VCI に対して30-50pt低い
- 認知リソースの消費を最小化することが真=美判定の精度向上に直結

月50判定 vs 月100判定:
- 浅い判定×多数 < 深い判定×少数（真=美の最小性）

#### 2. 案E の独立キュー方式との整合

Daiki が「軸」を選び、真=美判定 → 個別記事の判定は深く・少数が真=美

#### 3. 高リスク変更の事前判定

YMYL 領域で誤情報リスクが高い変更は方針段階で承認すべき。

### 高リスク変更5カテゴリ確定

```
方針段階で Daiki 判定が必要な変更:
  - タイトル変更
  - 構成大変更（h2 順序変更、セクション削除）
  - 法令引用の修正
  - 金利・限度額・料率の更新
  - LLM confidence: low の出力
```

### 棄却理由5カテゴリ確定

```
- 真=美違反（必然性なし）
- 事実誤認
- 規制違反
- Evidence不適切
- その他（自由記述）
```

棄却データは Step A-4 の3層学習ループ Layer2 に接続。

### UI 設計の概要（Phase 3 で詳細化）

差分パッチ承認画面の構造:
- 方針サマリ
- 差分1/N（Before/After、Rationale、Evidence）
- 承認/却下/編集承認ボタン
- 棄却時の理由カテゴリ選択
- 全差分適用後プレビュー機能

### 反証

**「案L で『方針が間違っていても差分まで生成される』のは無駄」**:
- 高リスク変更のみ案K 発動という運用で限定可能
- 通常リスクの記事では「差分を見れば方針の妥当性も同時判定」

**「案K の方が判定を分割できて良いのでは」**:
- ナレッジ1の「直感の片側照射」「ドメイン横断パターン認識」は深い判定で機能
- 浅い判定の繰り返しは Daiki の強みを活かさない
- → 案L が認知特性に合致

## 論点5: 既存システム（C系）との統合範囲

### 議論の出発点

5サブ論点に分解:
- 5-a: analysis_comments テーブルとの関係
- 5-b: 既存システムの改修可否
- 5-c: collectArticleContext の物理配置
- 5-c-2: 物理配置（リポジトリ構造）
- 5-d: 過去要因分析の活用

### Daiki の判定

```
5-a: 案Q（rewrite.db 内独立テーブル設計）
5-b: 推奨（MAX_CONTENT_CHARS 撤廃は別案件として保留）
5-c: 案S（shared/ 新規ディレクトリに切り出し）
5-c-2: 案c（s-tools/rewrite/ サブディレクトリ）
5-d: 案V（直近1件を「参考情報」として渡す）
```

### 案Q の妥当性（論点5-a）

```
理由:
  1. 案E 確定事項「目的別 DB 分離」と整合
  2. 要因分析（C系）と リライト指示書生成（案C）は別目的
  3. analysis_comments は monitor.db 配下に留め、rewrite.db は独立構造
```

新規テーブル:
- master_rewrite_session: セッション管理（モデル/コスト/状態）
- master_rewrite_diff: 個別差分（案H 構造、Daiki 判定結果）

### 案C範囲外の保留事項（論点5-b）

```
別案件として保留:
  既存 monitor-analysis.js の MAX_CONTENT_CHARS = 12000 撤廃
  
理由:
  - 既存要因分析は記事冒頭5-10%しか参照していない事実
  - これを修正するか否かは Daiki の判断対象
  - 案C で fetchWpStructured を実装すれば、それを既存 C系にも転用可能
  - ただし C系を改修すべきかは別判断
```

### 案S の妥当性（論点5-c）

```
理由:
  1. 案E 教訓「目的別」と「再発明禁止」のバランス
  2. データ集約（Layer 1）は目的非依存 → 共通化が真=美
  3. C系とリライトシステムが同じ集約ロジックを使う

物理配置:
  s-tools/node/shared/article-context.js
  s-tools/node/shared/wp-structured.js
  s-tools/node/shared/llm-adapters/
```

### 案c の妥当性（論点5-c-2）

```
s-tools/rewrite/ サブディレクトリ採用

理由:
  - 既存 s-tools リポジトリ内に rewrite/ サブディレクトリ追加が真=美
  - 新規リポジトリ分離は責任境界の物理化として Overkill
  - データ層（monitor.db / rewrite.db）は同一 data/ 配下で管理
```

### 案V の妥当性（論点5-d）

```
過去要因分析を「参考情報」として LLM に渡す

理由:
  1. 過去要因分析は「現状の課題仮説」として価値がある
     例: 「順位下落は競合の独自データ追加が原因」
     → リライトで何を直すべきかのヒントになる
  2. 古い分析の誤誘導リスクは限定的（直近1件のみ）
  3. トークンコスト無視できる

実装:
  buildRewritePrompt の Layer 2 に過去 analysis_comments 直近1件を含める
  「[注意: この分析は要因推定であり、リライト判断の絶対根拠ではない]」と明示
```

### 反証

**「案Q で別テーブル化すると、要因分析とリライト履歴の関連が見えなくなる」**:
- ATTACH DATABASE で同時参照可能（案E 確定）
- post_id を共通キーとして JOIN で追跡可能

**「案S の shared/ 切り出しは過剰設計では」**:
- C系とリライトの2系統で同じロジックを使う時点で抽象化価値あり
- 関数1個の切り出しは過剰設計とは言えない

**「案V で LLM の判断にバイアスがかかる」**:
- 「参考情報」と明示するプロンプト設計でバイアス低減
- Phase 4 実測で効果なし or 有害なら設定で OFF 可能

```
本文サイズ問題で正しかった処方:

  ✓ ノイズ除去（構造化抽出 = タグ間空白の累積を消す）
```

「足す」より「削る」が真=美。

### 2. 役割別の最適化

```
LLM 選定で正しかった処方:
  ✗ 単一モデル統一（Sonnet単独 or Opus単独）
  ✓ タスク別最適化（Opus 推論 + Sonnet 生成）

ベンチマーク差小 ≠ タスク特性差小
```

ベンチマークだけ見ると「Sonnet で十分」になるが、実タスクの推論密度を見ると Opus が必要な場面がある。

案E教訓の再適用:
