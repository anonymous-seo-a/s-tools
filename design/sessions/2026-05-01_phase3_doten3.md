## 3. 3-1: LLM 出力の事後検証フロー

### 検証タイミング

```
工程6'-A: Opus 4.7 分析 → 'awaiting_policy_judgment' or 'generating'
工程6'-B: Sonnet 4.6 差分生成 → 'generating'
★工程6'-C: Compliance 事後検証
工程6:  Daiki 真=美フィルタ
```

### rule_type 別の挙動

```
禁止表現   → risk_flag='regulation_violation' 設定 + Daiki 判定必須
必須表現   → 自動修正 (ng_text → correct_text)
正式表記   → 自動修正 (ng_text → correct_text)
```

理由:
- 禁止表現は文脈依存で自動修正不可（"審査が甘い" → 何に置換?）
- 必須/正式は置換可能、自動修正で Daiki 認知リソース節約

### 違反履歴の保存

```json
master_rewrite_diff.rationale.compliance = {
  "checked_at": "2026-05-01T...",
  "violations": [
    { "rule_id": 12, "rule_type": "禁止表現", "ng_text": "審査が甘い",
      "auto_fixed": false, "action": "risk_flag_set" },
    { "rule_id": 18, "rule_type": "必須表現", "ng_text": "電話なし",
      "auto_fixed": true, "before": "...", "after": "..." }
  ]
}
```

既存 rationale JSON 拡張（スキーマ変更不要、最小性◎）。

### 実装位置

`node/rewrite/llm-execution/compliance-checker.js`（新規、Phase 4）

## 4. 3-2: PR表記の存在検証

### 規制

景品表示法 ステマ規制（2023年10月1日施行）。
soico /no1/ は常時 PR 記事のため、サイト全体で固定 PR 表記が必要。

### 検証ロジック

```
入力: 全 diff 適用後の preview HTML
処理: 記事冒頭に「PR」「広告」「プロモーション」「アフィリエイト」
      「[PR]」「[広告]」のいずれかが存在するか正規表現検出
判定: 存在 → ok=true / 不在 → ok=false (警告)
```

### 実装位置

GET /api/rewrite/sessions/:session_id/preview レスポンス内に compliance_checks を含める。

WordPress テーマ側で全記事共通の PR 表記が固定実装されているか Phase 4 実装時に確認。
固定実装なら検証のみ、なしなら master_rules に必須ルール追加。

### 設計選択肢の評価

```

(A) rules拡張  ◎    | ✗     | ◎     | 構造混在 (テーブル単位最小性に釣られる罠)
(B) 新テーブル ◎    | ◎     | △     | 責務分離 (システム全体最小性 = 真=美)
```

警戒バイアス「テーブル単位の最小性 vs システム全体の最小性 取り違え」（案B で判明）対チェック:
**(A) はテーブル単位で釣られる罠**。システム全体では (B) が真=美。

### (B) 確定: master_ymyl_requirement 新規テーブル

```sql
CREATE TABLE master_ymyl_requirement (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  category TEXT NOT NULL,            -- 'cardloan' / 'cryptocurrency' / 'securities' / 'fx'
  requirement_name TEXT NOT NULL,    -- '実質年率' / '登録番号' 等
  detection_pattern TEXT NOT NULL,   -- 正規表現
  legal_basis TEXT,                  -- '貸金業法第15条' 等
  source_url TEXT,
  status TEXT NOT NULL DEFAULT 'draft',
  created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
  CHECK (status IN ('draft', 'verified', 'deprecated'))
);
CREATE INDEX idx_ymyl_category ON master_ymyl_requirement(category);
CREATE INDEX idx_ymyl_status ON master_ymyl_requirement(status);
```

### カテゴリ別必須項目 seed 案

```
cardloan (消費者金融、貸金業法第15条/第16条):
  実質年率 / 限度額 / 返済方式 / 登録番号

cryptocurrency (暗号資産、金商法・資金決済法):
  暗号資産交換業者登録番号 / 価格変動リスク開示 / 法定通貨ではない注記

securities (証券、金商法):
  金融商品取引業者登録番号 / 元本毀損リスク開示 / 手数料表示
  / 投資判断は自己責任注記

fx (FX、金商法):
  第一種金融商品取引業者登録番号 / レバレッジ規制（個人25倍）注記
  / 元本毀損リスク開示 / スワップ・スプレッド表示
```

正規表現パターンと法令根拠は Phase 4 実装時に Daiki + 法令確認で精査。
現時点では構造のみ確定。

### 検証ロジック

```
入力: 全 diff 適用後 preview HTML, post.category
処理:
  1. master_ymyl_requirement WHERE
       category = post.category AND status = 'verified'
  2. 各 requirement の detection_pattern で正規表現検出
  3. 全項目存在: OK / 不足項目あり: 警告 + missing[] にリストアップ

レスポンス:
{
  "compliance_checks": {
    "ymyl_required_items": {
      "ok": false,
      "category": "securities",
      "missing": ["元本毀損リスク開示"],
      "found": ["金融商品取引業者登録番号", "手数料表示", "投資判断は自己責任注記"]
    }
  }
}
```

### 不足検出時の挙動

MVP では警告表示のみ。自動補正（差分パッチ追加生成）は Phase 4 後に検討。

理由:
- 自動補正は文脈依存で誤補正リスク（YMYL 致命的）
- Daiki 目視判定で「忘れていた」を検出する装置として機能
- YMYL では人間最終承認が必須（案N 不採用原則と整合）

## 9. 反証プロトコル

### 反証1: 「LLM ベース検証の方が文脈理解で精度が高いのでは？」
master_rules は人間が運用整備した精密なルール集（92件 seed + 拡張）。
LLM 検証は確率的でハルシネーション混入リスク。YMYL では決定論的な正規表現が真=美。

### 反証2: 「禁止表現を自動修正できないのは制限では？」
自動修正の判断は文脈依存（"審査が甘い" → "審査基準は公表されていない" 等、複数候補）。
誤った自動修正は YMYL 致命的。Daiki 判定に回す方が安全。

### 反証3: 「(A) でも condition カラムで粒度区別すれば済むのでは？」
condition は「いつ適用するか」のフラグ。語彙ルールと構造要素チェックの本質的差を
condition で吸収するのは概念の歪曲。閉合性違反。

### 反証4: 「テーブル数を増やすのは最小性違反では？」
警戒バイアス「テーブル単位 vs システム全体」が判明済（案B で）。
システム全体での閉合性が真=美の上位原則。

### 反証5: 「不足検出時の自動補正をやらないなら検証の価値は？」
Daiki が「忘れていた」を検出する装置として機能。
YMYL では人間最終承認が必須（案N 不採用原則と整合）、自動補正よりも
「確認漏れ防止」の方が価値が高い。
