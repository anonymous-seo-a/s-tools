### 2-2. master_competitor_corpus 投入対象の決定

3案評価:
- (a) Daiki 指示通り 4 source_type (organic/paa/related_search/ai_overview_citation) 全投入
- (b) organic top N=3 のみ投入、他は serp_features 集計フラグ → 採用
- (c) ai_overview_citations の URL 持ちのみ追加で投入

判定: (b)
理由:
- knowledge/05 master_competitor_corpus 設計の本来用途は「競合 URL 収集」
- PAA は「質問テキスト」中心、URL 持ちは少数 → master_query_fanout 投入が自然 (案B # 4)
- related_searches は URL なしが多数 → master_query_fanout で generation_method='related_search'
- ai_overview_citations は実 SERP では稀 (本セッション smoke で 0 件)
- 警戒 [j] 取得対象範囲拡大バイアス対処

→ 案B # 4 (PAA / related_searches を master_query_fanout に投入) は本タスク範囲外、
  別タスクとして次セッション以降に encode

### 2-3. fact_set_snapshot プレースホルダの選定

3案評価:
- (a) NULL → schema NOT NULL 制約違反
- (b) 空文字列 '' → JSON パース fail リスク
- (c) JSON プレースホルダ '{"_pending":true}' → 採用

判定: (c)
理由:
- master_competitor_corpus.fact_set_snapshot 用途は JSON 想定
- '{"_pending":true}' は後段で master_fact_set 抽出 → UPDATE される設計
- 'pending' フラグで「未抽出状態」を識別可能、誤動作リスク低
- 別タスク (master_fact_set) で UPDATE 処理を実装する境界が明確

### 2-4. GSC page filter operator の選定

2案評価:
- (a) operator='equals' + 末尾 '/' あり / なし の両方試行
- (b) operator='contains' で末尾 '/' を吸収 → 採用

判定: (b)
理由:
- soico の URL 構造では post_id (4-5 桁数字) が他記事 URL に部分一致しない
- 末尾 / の有無を吸収する設計、コード簡潔性 ◎
- 反証検証: 誤検出リスクは構造的にゼロに近い (反証 3 で確認)

### 2-5. GSC 期間 days=90 の選定

3案評価:
- (a) days=28 → サンプル不足リスク
- (b) days=90 → 「最近」と「蓄積」のバランス → 採用 (デフォルト)
- (c) days=180 → 古いクエリパターン混入リスク

判定: (b)
理由:
- cardloan のような長尾分布クエリは 28 日では不足
- 180 日は記事の rewrite サイクルと整合しない (古い)
- 90 日は実装で smoke 引数 days で上書き可能、本格運用で再評価

→ 設計確定値ではなく、運用で調整可能な実装上の判断
